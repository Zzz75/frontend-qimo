## 1. 项目简介

**项目名称：** 基于 Vue3 的智能聊天助手系统

**项目背景：**
随着大模型技术的普及，智能聊天助手已广泛应用于学习、办公、编程辅助等场景。本项目以学习与实践现代前端工程化为目标，使用 Vue3 生态从零搭建了一个对话式 AI 助手 Web 应用。

**解决的问题：**

- 让用户无需后端账号体系，即可在浏览器中与大模型（DeepSeek API）实时对话。
- 解决传统问答"一次性返回、等待时间长"的体验问题，采用**流式逐字输出**。
- 解决多话题混杂的问题，提供**多会话管理**与**本地持久化**，刷新不丢失。

**主要功能：**

| 模块 | 功能说明 |
| --- | --- |
| 实时聊天 | 文本输入、Enter 发送、Shift+Enter 换行、自动滚动到底部 |
| AI 回复 | 调用大模型接口，支持加载状态、异常提示 |
| 流式输出 | 基于 SSE 协议逐字渲染回复，体验接近 ChatGPT |
| 会话管理 | 新建 / 切换 / 删除会话，标题自动生成 |
| 数据持久化 | LocalStorage 存储会话与消息，刷新不丢失 |
| Markdown 渲染 | 标题、列表、表格、引用、链接等富文本解析 |
| 代码高亮 | 多语言语法高亮（JS / HTML / CSS / Vue / Python 等）|
| 响应式适配 | 桌面端双栏布局，移动端抽屉式侧边栏 |

**技术亮点（一句话概括，由成员 A 讲）：**
全程 **TypeScript + Composition API**，Pinia 按职责拆分为多个 Store，**手写 SSE 流式解析**，并在 Markdown 渲染层做了 **XSS 安全防护**。

## 4. 每位成员详细讲稿提纲

### 4.1 成员 A 讲稿（架构与技术，约 5.5 分钟）

**【开场白】**
> 各位老师好，我负责系统的架构与技术实现。我从工程结构、状态管理、接口与流式实现、安全与优化几个方面，介绍这个项目背后是怎么运转的。

**【讲解顺序】**

1. 系统架构与目录分层（约 1.5 分钟）
   - 分层：`views`（页面）→ `components`（组件）→ `stores`（状态）→ `composables`（逻辑复用）→ `api`（接口）→ `utils`（工具）→ `types`（类型）。
   - **职责清晰、单向数据流**：组件触发 Action，Store 修改状态，视图响应式更新。
   - 组件按 `chat / session / common` 三类拆分，职责单一、通过 Pinia 共享状态，避免 props 层层透传。

2. 状态管理设计（约 1 分钟）
   - Pinia 按职责拆成多个 Store：
     - `app`：模型名、接口配置；
     - `ui`：主题、侧边栏、全局 loading；
     - `session`：会话列表、当前会话、增删切换；
     - `chat`：消息列表、发送 / 流式 / 错误状态（数据流核心枢纽）。
   - 采用 **setup 函数式写法**，与 Composition API 风格统一，类型推导更好。

3. 接口封装与流式响应（约 1.5 分钟）—— **核心亮点**
   - `api/client`：基于 Axios 创建实例，**请求 / 响应拦截器**统一注入鉴权头、请求 ID，并统一错误结构。
   - `api/chat`：因 Axios 不便处理流式，改用 **`fetch` + `ReadableStream`** 手动读取，**`TextDecoder` 解码**，按 SSE 规范解析 `data:` 行、识别 `[DONE]` 与 `finish_reason` 结束信号，逐块回调实现逐字渲染。

4. 难点、安全与优化（约 1.5 分钟）
   - **SSE 分包问题**：网络分片可能切断一个事件，用 buffer 累积、按 `\n\n` 切分，未完整的事件留到下一轮。
   - **发送失败回滚**：流式中断 / 报错时移除空的助手消息，恢复可发送状态。
   - **XSS 安全**：Markdown 渲染**关闭原始 HTML 渲染**（`renderer.html = () => ''`），属性做转义，链接强制 `http/https` 校验。
   - **持久化校验**：从 LocalStorage 读取时过滤脏数据，JSON 解析全程 try-catch 兜底。
   - **性能**：路由懒加载、流式**增量追加**而非整体替换、`computed` 派生当前会话消息。

**【结束语】**
> 总体来说，这个项目让我们完整实践了 Vue3 工程化、状态管理、接口封装和流式交互。不足是暂未接入后端账号体系与云端同步，后续计划增加用户登录、多端同步等。感谢各位老师，欢迎提问。

## 6. 项目技术亮点（主要由成员 A 讲述）

### 6.1 Vue3 Composition API + `<script setup>`
全项目组件均使用 `<script setup lang="ts">`，逻辑按功能聚合，配合 TypeScript 获得完整类型提示；通用逻辑抽成组合式函数（`useMarkdown`、`useHighlight`、`useAutoScroll`）实现跨组件复用。

### 6.2 组件封装
按 `chat / session / common` 三大类划分，每个组件职责单一，通过 Pinia 共享状态，避免 props 层层透传。

### 6.3 状态管理（Pinia）
按职责拆分 `app / ui / session / chat` 多个 Store，使用 setup 函数式写法。`chat` Store 集中管理消息、发送态、流式态与错误态，是数据流核心枢纽。

### 6.4 路由设计（Vue Router）
采用 `createWebHistory` 模式，配置 `/`（聊天）与 `/:pathMatch(.*)*`（404）两条路由，使用 `() => import()` **懒加载**，减小首屏体积。

### 6.5 接口封装（Axios + Fetch 双通道）
普通请求走 Axios 实例（统一拦截器注入鉴权、统一错误处理）；流式请求走原生 `fetch + ReadableStream`，手写 SSE 解析。两者通过 `getAuthHeaders` 复用鉴权逻辑。

### 6.6 流式输出（SSE，核心难点）
`fetch` 拿到 `response.body.getReader()`，循环 `read()`，`TextDecoder` 增量解码，按 `\n\n` 切分事件、解析 `data:`、识别 `[DONE]` 与 `finish_reason`，逐块回调 `onChunk` 实现逐字渲染。

### 6.7 安全防护
Markdown 渲染**关闭原始 HTML 渲染**（`renderer.html = () => ''`），属性做 HTML 转义，链接强制协议校验并加 `rel="noopener noreferrer"`，防止 XSS；API Key 通过环境变量注入，不硬编码。

### 6.8 性能与健壮性
路由懒加载、流式增量追加、`computed` 派生当前会话消息；`utils/storage` 读写带类型校验，JSON 解析全程 try-catch 兜底，刷新可靠恢复。

### 7.2 成员 A 主答的"技术问题"

**Q1：为什么选 Vue3 而不是 Vue2 / React？**
> Vue3 的 Composition API 更利于逻辑复用与 TypeScript 集成，响应式基于 Proxy 性能更好，`<script setup>` 写法简洁；相比 React 上手成本更低、模板更直观。

**Q2：为什么用 Pinia 而不是 Vuex？**
> Pinia 是 Vue 官方新一代状态管理库：去掉了 mutations，API 更简洁；对 TS 是一等公民；天然支持 setup 写法；模块化无需嵌套 modules。

**Q3：项目最大的难点是什么？**
> 流式响应（SSE）的解析。Axios 不便处理流，改用 fetch 的 ReadableStream 手动读取。难点在网络分包会切断完整事件，用 buffer 累积、按 `\n\n` 分割、不完整片段留到下一轮，并正确识别 `[DONE]` 和 `finish_reason`。

**Q4：流式为什么用 fetch 而不是 Axios？**
> Axios 默认基于 XMLHttpRequest，对流式支持不友好，拿不到边到达边处理的可读流；fetch 的 `response.body` 是 ReadableStream，可逐块读取。普通请求仍用 Axios 以复用拦截器。

**Q5：数据怎么持久化的？刷新会丢吗？**
> 用 LocalStorage 分键存储会话、消息、UI 偏好等，状态变更后写入，启动时读取并做类型校验过滤脏数据，所以刷新后会话和记录都能恢复。

**Q6：如何防止 XSS？**
> AI 返回内容经 marked 渲染为 HTML 有注入风险，我们在自定义 renderer 中关闭原始 HTML 渲染、对属性做转义、对链接强制协议校验；API Key 也走环境变量不写进代码。

**Q7：怎么优化性能？**
> 路由与组件懒加载减小首屏；流式回复增量追加避免整段重渲染；用 `computed` 派生当前会话消息；持久化只在状态变更时写入。

**Q8：怎么做组件 / 逻辑复用？**
> 组件按职责拆分 + Pinia 共享状态复用；逻辑用组合式函数（`useMarkdown`、`useHighlight`、`useAutoScroll`）抽离，多组件按需引入。

**Q9：发送失败怎么办，状态会乱吗？**
> 设计了回滚机制：流式中断 / 报错时移除内容为空的助手占位消息、重置发送与流式状态，并用 try/finally 保证发送态一定复位。

**Q10：TypeScript 起了什么作用？**
> 为消息、会话、API 请求/响应等定义类型（`types/` 目录），编译期发现类型错误，配合 Pinia 和 Composition API 获得完整类型推导，降低运行时 bug。
