# 《基于 Vue3 的智能聊天助手系统》答辩方案

> 适用形式：两人轮流答辩 ｜ 总时长：10 分钟左右 ｜ 每人约 5 分钟
> 技术栈：Vue 3（Composition API + `<script setup>`）、Vite、Pinia、Vue Router、Axios、TypeScript、marked、highlight.js

---

## 1. 项目简介

**项目名称：** 基于 Vue3 的智能聊天助手系统

**项目背景：**
随着大模型技术的普及，智能聊天助手已广泛应用于学习、办公、编程辅助等场景。本项目以学习与实践现代前端工程化为目标，使用 Vue3 生态从零搭建了一个对话式 AI 助手 Web 应用。

**解决的问题：**

- 让用户无需后端账号体系，即可在浏览器中与大模型（DeepSeek API）实时对话。
- 解决传统问答“一次性返回、等待时间长”的体验问题，采用**流式逐字输出**。
- 解决多话题混杂的问题，提供**多会话管理**与**本地持久化**，刷新不丢失。

**主要功能：**

| 模块 | 功能说明 |
| --- | --- |
| 实时聊天 | 文本输入、Enter 发送、Shift+Enter 换行、自动滚动到底部 |
| AI 回复 | 调用大模型接口，支持加载状态、异常提示、失败重试 |
| 流式输出 | 基于 SSE 协议逐字渲染回复，体验接近 ChatGPT |
| 会话管理 | 新建 / 切换 / 删除会话，标题自动生成 |
| 数据持久化 | LocalStorage 存储会话与消息，刷新不丢失 |
| Markdown 渲染 | 标题、列表、表格、引用、链接等富文本解析 |
| 代码高亮 | 多语言语法高亮（JS / HTML / CSS / Vue / Python 等）|
| 主题切换 | 亮色 / 暗色一键切换，偏好持久化 |
| 角色切换 | 可配置多个 AI 角色（系统提示词）|
| 响应式适配 | 桌面端双栏布局，移动端抽屉式侧边栏 |

**技术亮点（一句话概括）：**
全程 **TypeScript + Composition API**，Pinia 按职责拆分为 4 个 Store，**手写 SSE 流式解析**，并在 Markdown 渲染层做了 **XSS 安全防护**。

**【开场白】**

**【讲解顺序】**

1. 系统架构与目录分层（约 1.5 分钟）
   - 分层：`views`（页面）→ `components`（组件）→ `stores`（状态）→ `composables`（逻辑复用）→ `api`（接口）→ `utils`（工具）→ `types`（类型）。
   - **职责清晰、单向数据流**：组件触发 Action，Store 修改状态，视图响应式更新。

2. 状态管理设计（约 1 分钟）
   - Pinia 按职责拆成 4 个 Store：
     - `app`：AI 角色、模型名、接口配置；
     - `ui`：主题、侧边栏、全局 loading；
     - `session`：会话列表、当前会话、增删切换；
     - `chat`：消息列表、发送 / 流式 / 错误状态。
   - 采用 **setup 写法（函数式）**，与 Composition API 风格统一，类型推导更好。

3. 接口封装与流式响应（约 1.5 分钟）—— **核心亮点**
   - `api/client`：基于 Axios 创建实例，**请求 / 响应拦截器**统一注入鉴权头、`X-Request-Id`，并通过 `normalizeApiError` 统一错误结构。

4. 难点与优化（约 1 分钟）
   - **SSE 分包问题**：网络分片可能切断一个事件，用 buffer 累积、按 `\n\n` 切分，未完整的事件留到下一轮。
   - **发送失败回滚**：流式中断时移除空的助手消息，恢复可发送状态，并支持「重试上一条」。
   - **持久化校验**：从 LocalStorage 读取时用 `isValidSession / isValidMessage` 过滤脏数据，保证健壮性。
   - **性能优化**：路由懒加载、流式增量更新（只追加内容不整体替换）、`computed` 派生当前会话消息。

**【重点说明内容】**

- 为什么流式要用 fetch 而不是 Axios。
- SSE 数据如何一步步从二进制 → 文本 → JSON → 增量内容。
- 数据持久化策略与异常兜底，体现工程严谨性。

## 5. 项目技术亮点（可直接讲述版）

### 5.1 Vue3 Composition API + `<script setup>`
全项目组件均使用 `<script setup lang="ts">`，逻辑按功能聚合而非按选项拆散，配合 TypeScript 获得完整类型提示；通用逻辑抽成组合式函数（`useMarkdown`、`useHighlight`、`useAutoScroll`、`useStreamResponse`）实现跨组件复用。

### 5.2 组件封装
按 `chat / session / common` 三大类划分：

- 聊天类：`ChatInput`、`MessageList`、`MessageItem`、`MarkdownRenderer`、`TypingIndicator`；
- 会话类：`SessionSidebar`、`SessionItem`、`NewSessionButton`、`SessionDeleteDialog`；
- 通用类：`AppHeader`、`LoadingState`、`ErrorToast`、`EmptyState`。
每个组件职责单一，通过 Pinia 共享状态，避免 props 层层透传。

### 5.3 状态管理（Pinia）
按职责拆分 `app / ui / session / chat` 四个 Store，使用 setup 函数式写法。`chat` Store 集中管理消息、发送态、流式态与错误态，是数据流的核心枢纽。

### 5.4 路由设计（Vue Router）
采用 `createWebHistory` 模式，配置 `/`（聊天）、`/settings`（设置）、`/:pathMatch(.*)*`（404）三条路由，全部使用 `() => import()` **懒加载**，减小首屏体积。

### 5.5 接口封装（Axios + Fetch 双通道）
- 普通请求走 Axios 实例，统一拦截器注入鉴权、统一错误处理；
- 流式请求走原生 `fetch + ReadableStream`，手写 SSE 解析。
两者通过 `getAuthHeaders` 复用鉴权逻辑，通过 `normalizeApiError` 统一错误格式。

### 5.6 流式输出（SSE，核心难点）
`fetch` 拿到 `response.body.getReader()`，循环 `read()`，`TextDecoder` 增量解码，按 `\n\n` 切分事件、解析 `data:`、识别 `[DONE]` 与 `finish_reason`，逐块回调 `onChunk` 实现逐字渲染。

### 5.7 安全与权限控制
- Markdown 渲染时 **关闭原始 HTML 渲染**（`renderer.html = () => ''`），并对属性做转义（`escapeHtmlAttribute`），有效防止 **XSS 注入**；
- 链接强制 `http/https` 协议校验并加 `rel="noopener noreferrer"`；
- 接口密钥通过环境变量（`.env.local`）注入，不硬编码进源码。

### 5.8 性能优化
- 路由与视图懒加载，降低首屏体积；
- 流式回复**增量追加**而非整体替换，减少重渲染；
- 当前会话消息用 `computed` 派生，避免重复计算；
- 高频输入交互通过禁用态控制，避免重复发送。

### 5.9 持久化与健壮性
`utils/storage` 封装统一的读写接口，读取时对会话与消息做类型校验，过滤脏数据；JSON 解析全部 try-catch 兜底，保证刷新后数据可靠恢复。

## 6. 老师可能提问的问题与标准回答

**Q1：为什么选择 Vue3 而不是 Vue2 / React？**
> Vue3 的 Composition API 更利于逻辑复用与 TypeScript 集成，响应式系统基于 Proxy 性能更好，且 `<script setup>` 写法简洁。相比 React，Vue 的上手成本更低、模板语法更直观，适合本项目快速迭代。

**Q2：Pinia 和 Vuex 的区别？为什么用 Pinia？**
> Pinia 是 Vue 官方新一代状态管理库。相比 Vuex：① 去掉了 mutations，只有 state / getters / actions，API 更简洁；② 对 TypeScript 支持是一等公民，类型推导更好；③ 天然支持 Composition API 的 setup 写法；④ 支持模块化且无需嵌套 modules。所以我们选择 Pinia。

**Q3：项目中遇到的最大难点是什么？**
> 流式响应（SSE）的解析。Axios 不便处理流，我们改用 fetch 的 ReadableStream 手动读取。难点在于网络分包会切断一个完整事件，我们用 buffer 累积、按 `\n\n` 分割、把不完整的片段留到下一轮，并正确识别 `[DONE]` 和 `finish_reason` 结束信号。

**Q4：如何进行组件复用？逻辑复用又是怎么做的？**
> 组件层面通过职责拆分 + Pinia 共享状态实现复用（如 `MessageItem`、`SessionItem`）。逻辑层面用组合式函数（Composables）抽离，如 `useMarkdown`、`useHighlight`、`useAutoScroll`，在多个组件中按需引入，实现无副作用的逻辑复用。

**Q5：如何优化页面性能？**
> ① 路由与组件懒加载减小首屏体积；② 流式回复采用增量追加，避免整段重渲染；③ 用 `computed` 派生当前会话消息，避免重复计算；④ 持久化只在状态变更时写入。

**Q6：数据是怎么持久化的？刷新会丢吗？**
> 使用 LocalStorage 分键存储（会话、消息、UI 偏好、应用配置）。每次状态变更后写入，启动时读取并做类型校验过滤脏数据，所以刷新后会话和聊天记录都能恢复，不会丢失。

**Q7：如何防止 XSS 等安全问题？**
> AI 返回内容会经 marked 渲染为 HTML，存在注入风险。我们在自定义 renderer 中**关闭了原始 HTML 渲染**，对代码块语言、链接标题等属性做 HTML 转义，并对链接强制协议校验。API Key 也通过环境变量注入，不写进代码。

**Q8：流式输出为什么不用 Axios，而用 fetch？**
> Axios 默认基于 XMLHttpRequest，对流式（chunked / SSE）支持不友好，拿不到边到达边处理的可读流。fetch 的 `response.body` 是 ReadableStream，可以用 reader 逐块读取，更适合流式场景。普通请求我们仍用 Axios 以复用拦截器。

**Q9：发送消息失败了怎么办？如何保证状态不混乱？**
> 我们设计了回滚机制：流式中断或报错时，移除内容为空的助手占位消息，重置发送 / 流式状态，并记录最后一条用户输入，提供「重试」入口。同时用 try / finally 保证 `isSending` 一定会被复位。

**Q10：项目里 TypeScript 起到了什么作用？**
> 全项目用 TS，为消息、会话、API 请求 / 响应等定义了类型（`types/` 目录），在编译期就能发现类型错误；配合 Pinia 和 Composition API 获得完整的类型推导和编辑器提示，显著降低了运行时 bug。

**Q11：主题切换是怎么实现的？**
> 通过在 `document.documentElement` 上设置 `data-theme` 属性，CSS 中用变量（`--color-*`）定义两套配色，切换时只改属性，浏览器自动应用对应变量，并把偏好写入 LocalStorage 持久化。

**Q12：会话标题是怎么生成的？**
> 新建会话默认标题为「新会话」，当用户发出第一条消息时，取消息内容前若干字符（超出截断加省略号）作为标题，提升列表可读性。

**如何体现个人贡献：**

- 成员 A 强调「我负责的交互组件 / 渲染 / 体验」，成员 B 强调「我负责的架构 / 接口 / 流式」。
- 讲到自己模块时用「我在实现 XX 时遇到 YY，通过 ZZ 解决」的句式，突出独立思考。
- 回答提问时，本人负责的模块由本人主答，体现分工清晰、各有所长。
