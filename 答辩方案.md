## 1. 项目简介

**项目名称：** 基于 Vue3 的智能聊天助手系统

**项目背景：**
随着大模型技术的普及，智能聊天助手已广泛应用于学习、办公、编程辅助等场景。本项目以学习与实践现代前端工程化为目标，使用 Vue3 生态从零搭建了一个对话式 AI 助手 Web 应用。

**解决的问题：**

- 让用户无需后端账号体系，即可在浏览器中与大模型（DeepSeek API）实时对话。
- 解决传统问答"一次性返回、等待时间长"的体验问题，采用**流式逐字输出**。
- 解决多话题混杂的问题，提供**多会话管理**与**本地持久化**，刷新不丢失。

**主要功能：**

| 模块 | 功能说明 |
| --- | --- |
| 实时聊天 | 文本输入、Enter 发送、Shift+Enter 换行、自动滚动到底部 |
| AI 回复 | 调用大模型接口，支持加载状态、异常提示 |
| 流式输出 | 基于 SSE 协议逐字渲染回复，体验接近 ChatGPT |
| 会话管理 | 新建 / 切换 / 删除会话，标题自动生成 |
| 数据持久化 | LocalStorage 存储会话与消息，刷新不丢失 |
| Markdown 渲染 | 标题、列表、表格、引用、链接等富文本解析 |
| 代码高亮 | 多语言语法高亮（JS / HTML / CSS / Vue / Python 等）|
| 响应式适配 | 桌面端双栏布局，移动端抽屉式侧边栏 |

**技术亮点（一句话概括，由成员 A 讲）：**
全程 **TypeScript + Composition API**，Pinia 按职责拆分为多个 Store，**手写 SSE 流式解析**，并在 Markdown 渲染层做了 **XSS 安全防护**。

## 4. 每位成员详细讲稿提纲

### 4.1 成员 A 讲稿（架构与技术，约 5.5 分钟）

**【开场白】**
> 各位老师好，我负责系统的架构与技术实现。我从工程结构、状态管理、接口与流式实现、安全与优化几个方面，介绍这个项目背后是怎么运转的。

**【讲解顺序】**

1. 系统架构与目录分层（约 1.5 分钟）
   - 分层：`views`（页面）→ `components`（组件）→ `stores`（状态）→ `composables`（逻辑复用）→ `api`（接口）→ `utils`（工具）→ `types`（类型）。
   - **职责清晰、单向数据流**：组件触发 Action，Store 修改状态，视图响应式更新。
   - 组件按 `chat / session / common` 三类拆分，职责单一、通过 Pinia 共享状态，避免 props 层层透传。

2. 状态管理设计（约 1 分钟）
   - Pinia 按职责拆成多个 Store：
     - `app`：模型名、接口配置；
     - `ui`：主题、侧边栏、全局 loading；
     - `session`：会话列表、当前会话、增删切换；
     - `chat`：消息列表、发送 / 流式 / 错误状态（数据流核心枢纽）。
   - 采用 **setup 函数式写法**，与 Composition API 风格统一，类型推导更好。

3. 接口封装与流式集成（约 1.5 分钟）—— **核心亮点**
   - `api/client`：基于 Axios 创建实例，**请求 / 响应拦截器**统一注入鉴权头、请求 ID，并统一错误结构。
   - `api/chat`：因 Axios 不便处理流式，改用 **`fetch` + `ReadableStream`** 手动读取、**`TextDecoder` 解码**，把数据流接进来。
   - **接力衔接**：「拿到数据流之后**怎么按帧拆包、拼半包**，是我搭档刚才讲的那部分」——拿到内容后通过回调逐块送进 Vue 响应式渲染，实现逐字输出。

4. 难点、安全与优化（约 1.5 分钟）
   - **发送失败回滚**：流式中断 / 报错时移除空的助手消息，恢复可发送状态。
   - **XSS 安全**：Markdown 渲染**关闭原始 HTML 渲染**（`renderer.html = () => ''`），属性做转义，链接强制 `http/https` 校验。
   - **增量渲染**：流式回复**只追加不整体替换**，配合 Vue 响应式减少重渲染。
   - **性能**：路由懒加载、`computed` 派生当前会话消息、`nextTick` 后再自动滚动。

**【结束语】**
> 总体来说，这个项目让我们完整实践了 Vue3 工程化、状态管理、接口封装和流式交互。不足是暂未接入后端账号体系与云端同步，后续计划增加用户登录、多端同步等。感谢各位老师，欢迎提问。

**【数据解析与存储逻辑】（约 1 分钟，加分段，结合嵌入式背景照念）**

> 这些功能里，我重点参与的是**数据层的处理**。我是嵌入式 / C 语言方向的，刚开始接触前端，但**数据怎么解析、怎么存储，本质和我做嵌入式是相通的**，所以这块由我负责。
> 第一是**流式数据的分包与缓冲**。AI 的回复不是一次性返回的，而是像数据流一样**一段一段到达**，而且**一个完整的消息可能被网络切成两半**。这跟我以前做嵌入式时**解析串口（UART）数据帧**遇到的问题一模一样——数据是连续字节流，一帧可能分两次收到。我们的处理办法也一样：**用一个缓冲区（buffer）把收到的数据先攒着，按分隔符（两个换行 `\n\n`）切分出一个个完整事件，最后那段如果不完整，就留着和下一批数据拼起来**，再判断 `[DONE]` 这种结束标志，确保每条消息都拼完整。
> 第二是**本地存储**。聊天记录用浏览器的 LocalStorage 存，它是**键值对（key-value）**存储，只能存字符串。所以存的时候把数据**序列化成 JSON 字符串**，读的时候再**解析回来**，并且**读不到就给一个默认空值兜底**，避免出错。这和嵌入式里往 Flash / EEPROM 里按地址存结构体、读出来再还原是一个道理。
> 此外还有几个工具函数，比如用 `crypto.randomUUID()` 给每条消息和会话生成**唯一 ID**（类似设备唯一编号），以及时间戳格式化。

## 6. 项目技术亮点（主要由成员 A 讲述）

### 6.1 Vue3 Composition API + `<script setup>`
全项目组件均使用 `<script setup lang="ts">`，逻辑按功能聚合，配合 TypeScript 获得完整类型提示；通用逻辑抽成组合式函数（`useMarkdown`、`useHighlight`、`useAutoScroll`）实现跨组件复用。

### 6.2 组件封装
按 `chat / session / common` 三大类划分，每个组件职责单一，通过 Pinia 共享状态，避免 props 层层透传。

### 6.3 状态管理（Pinia）
按职责拆分 `app / ui / session / chat` 多个 Store，使用 setup 函数式写法。`chat` Store 集中管理消息、发送态、流式态与错误态，是数据流核心枢纽。

### 6.4 路由设计（Vue Router）
采用 `createWebHistory` 模式，配置 `/`（聊天）与 `/:pathMatch(.*)*`（404）两条路由，使用 `() => import()` **懒加载**，减小首屏体积。

### 6.5 接口封装（Axios + Fetch 双通道）
普通请求走 Axios 实例（统一拦截器注入鉴权、统一错误处理）；流式请求走原生 `fetch + ReadableStream`，手写 SSE 解析。两者通过 `getAuthHeaders` 复用鉴权逻辑。
